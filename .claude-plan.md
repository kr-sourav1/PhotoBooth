# PhotoBooth — Preview-Based Photo Selection Platform (B2B SaaS)

## Context

Photography studios shoot thousands of high-res images per event (8 GB+ per shoot).
Uploading all originals to the cloud so clients can pick favorites is slow, expensive,
and wasteful — clients only ever choose a small subset. This project solves that with a
**preview-first** workflow: only lightweight previews (~200 KB vs ~5 MB) go to the cloud;
**original high-res files never leave the studio's machine**. Clients browse previews and
select; a studio-side desktop app then collects the matching originals locally.

This is being built as a **product to sell B2B** to studios, so it is designed as a
**multi-tenant SaaS** (many studios sign up, isolated accounts, subscription billing) with
a polished, installable **desktop app** for studios and a **responsive web/PWA gallery**
for their clients.

### Confirmed decisions
- **Studio tool:** cross-platform **desktop app** (no Python install for the end user).
- **Business model:** **multi-tenant SaaS**, subscription-billed.
- **Client access:** **responsive web app / PWA** via share link (no client login required).
- **Cloud:** **Supabase** (Postgres + Auth) + **Cloudflare R2** object storage for previews.

## System Architecture — three components

```
┌──────────────────────────┐         ┌─────────────────────────┐         ┌──────────────────────────┐
│  STUDIO DESKTOP APP       │         │  CLOUD BACKEND          │         │  CLIENT GALLERY (Web/PWA)│
│  (runs where 8GB live)    │         │  (Supabase + R2)        │         │  (share link, no login)  │
│                           │ upload  │                         │  serve  │                          │
│  1. Pick originals folder │ previews│  Postgres (multi-tenant)│ previews│  Browse previews         │
│  2. Generate previews ────┼────────▶│  Auth (studios)         ├────────▶│  Favorite / select       │
│  3. Local SQLite manifest │         │  R2 preview storage+CDN │         │  Submit selection        │
│  4. Auto-collect selected │◀────────┤  Selections table       │◀────────┤                          │
│     originals → folder    │ selection│  Stripe billing         │ writes  │                          │
└──────────────────────────┘  list   └─────────────────────────┘         └──────────────────────────┘
        ▲ originals stay 100% local                                  clients never see originals
```

### End-to-end data flow
1. Studio signs in to desktop app → creates a **Project** (event) → gets `project_id`.
2. Desktop app scans the chosen originals folder, generates previews **locally and in
   parallel**, and builds a **local SQLite manifest** mapping each photo's `uuid` →
   absolute original path + original filename + content hash.
3. Only previews upload to **R2**; one `photo` row per image is written to Postgres
   (`uuid`, original filename, preview URL, dimensions, project_id). **Originals never upload.**
4. Studio shares a tokenized link. Client opens the **web gallery**, browses previews,
   selects favorites, and submits → selection stored in Postgres.
5. Desktop app subscribes to selection status; on submit it pulls the selected `uuid` list,
   matches against the local manifest, and **copies matched originals into a
   `Selected Photos/` folder** — fully automated.

### Key design improvement over the naive spec
The original plan matches purely on **filename**, but filenames collide in real shoots
(two cameras both produce `IMG_1023.jpg`). We assign each photo a **stable UUID** stored in
both the cloud record and the local SQLite manifest, and match on **UUID first, filename as
fallback**. The manifest also stores a content hash so re-collection still works if folders
are moved or renamed. This makes the automation reliable at the scale studios actually shoot.

## Recommended Tech Stack

### Studio Desktop App
- **Tauri 2** (Rust shell + web frontend) — small signed installers, built-in **auto-updater**,
  far lighter/more secure than Electron → better for a sellable B2B product.
- **Image processing in Rust:** `image` + `fast_image_resize` + `rayon` (parallel previews
  across CPU cores) + `kamadak-exif` (honor EXIF orientation). Thousands of previews in
  seconds. *Optional preview **watermarking** (studios want this so clients can't reuse
  low-res previews without paying).*
- **Local manifest:** SQLite via `rusqlite`.
- **UI:** React + TypeScript + Tailwind (shared components with the web gallery).
- *Alternative if the team prefers all-JavaScript: Electron + `sharp` (libvips). Noted, but
  Tauri is the recommendation for binary size, security, and distribution polish.*

### Cloud Backend
- **Supabase:** Postgres (multi-tenant data), **Auth** (studio accounts), **Row Level
  Security** for tenant isolation, **Edge Functions** (Deno) for server logic
  (issue share tokens, finalize selections, R2 signed-upload URLs, Stripe webhooks).
- **Cloudflare R2** for preview storage — **S3-compatible with zero egress fees**, served via
  Cloudflare CDN. Directly attacks the "clients browsing = lots of reads = high cost" problem.
- **Stripe** for B2B subscriptions (plan tiers gated on storage / active projects).

### Client Gallery (Web / PWA)
- **React + TypeScript + Vite + Tailwind**, **TanStack Query** for data, `vite-plugin-pwa`
  for installable/offline-friendly browsing. Token-based share links (no client signup).

### Shared / tooling
- **pnpm + Turborepo monorepo**; shared `ui` (components) and `types` packages; Supabase-
  generated TypeScript types as the single source of truth across desktop, web, and functions.
- TypeScript everywhere, ESLint/Prettier, **Vitest** (unit) + **Playwright** (gallery E2E),
  **GitHub Actions** CI, Tauri auto-update + code signing (Apple notarization + Windows cert).

## Data model (core tables, all tenant-scoped via RLS on `studio_id`)
- `studios` — tenant (name, plan, Stripe customer id, branding).
- `users` — studio members (auth), FK `studio_id`, role.
- `projects` — an event/shoot (name, client name, share_token, status, expiry).
- `photos` — `uuid`, `project_id`, `original_filename`, `preview_path` (R2), width/height, hash.
- `clients` / `selections` — selected `photo_uuid`s per project, submit timestamp, status.
- `subscriptions` — Stripe plan/state mirror for entitlement checks.

## Cross-cutting concerns
- **Security:** RLS for tenant isolation; signed/expiring R2 URLs; tokenized client links
  with expiry; Stripe webhook signature verification; code-signed desktop binaries for trust.
- **Cost control (the whole point):** R2 zero-egress + CDN for preview reads; previews only;
  plan-based storage caps; auto-expire/cleanup old projects.
- **Reliability:** resumable/chunked preview uploads; UUID+hash matching; manifest survives
  folder moves; idempotent selection finalize.
- **Scale:** previews are tiny and reads are CDN-cached, so the backend stays cheap as studios
  and clients grow; heavy CPU work (preview gen) is offloaded to the studio's own machine.

## Verification (per phase, end-to-end)
1. **Preview engine (Phase 1):** point the desktop app at a folder of 1000+ high-res images;
   confirm previews are correct/oriented, ~200 KB each, manifest rows = image count.
2. **Round trip (Phases 2–4):** generate + upload previews → open gallery link on a phone →
   select N photos → submit → confirm desktop app produces a `Selected Photos/` folder
   containing exactly those N originals (verify by UUID, including a deliberate filename
   collision across two subfolders).
3. **Tenant isolation (Phase 0/5):** sign in as Studio A, confirm Studio B's projects/photos
   are invisible via API and RLS.
4. **Billing (Phase 5):** Stripe test-mode subscribe/cancel; confirm plan limits enforce
   storage/project caps.
5. **Automated tests:** Vitest for selection-matching logic; Playwright for the gallery
   select-and-submit flow; CI green on GitHub Actions.
